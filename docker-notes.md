## Докер, тут все своими словами 

**docker --version** - чекнуть версию  
**sudo docker build -t <имя> <путь>** - читает файл *(Dockerfile с параметрами запуска для докера который надо создать ранее)* и собирает из него готовый шаблон - образ с именем <name>  
**sudo docker run -d --name <имя контейнера> <имя образа из docker build -t>** - Берет образ <имя> который мы создали через `sudo docker build -t`  
**sudo docker ps** - посмотреть список всех работающих в данный момент контейнеров, их айди и скольки они уже запущенны   
**sudo docker logs <имя>** - выводит на экран все что код в приложении который я запустил в контейнере успело отобразить  
**sudo docker exec -it <имя> bash** - пробивает изоляцию и подключает терминал `-it` к командной оболчке `bash` внутри работающего контейнера   

## Docker Compose своими словами

`Короче в кратзе, как я понял docker compose это по сути мод для докера. то есть, ты скачиваешь его и благодаря нему запускаешь процесс поверх своего докера, в котором уже все натсроено в одном файле. это как пуьт упралвения, получается ты создаешь папку `например docker_test` и там создаешь разумеется файл Dockerfile чтобы докер знал как запустить проекта также какой нибудь скрипт например `main.py`. Затем, установив docker compose ты в этой же папке создаешь файл с расширением .yml , в котором ты можешь указать имя проекта, дать путь на скрипт `main.py` , подключить дата базу и т.д. То есть по сути ты вместо того тчобы настраивать docker build с кучей флагов и открыть потом еще другие консоли отдельно для датабазы и т.д. просто устанавливаешь docker compose и создаешь один файл с расширением `.yml` в котором натсраиваешь все как нужно.`   

**Главное отличие докера от докер композа :**  

* **Docker** отвечает за создание одного конкретного изолированного контейнера(коробки). Он знает как упаковать пайтон-скрипт  

* **Docker compose** отвечает за оркестрацию(пульт управление) - то есть за одновременный запуск и связь целой пачки контейнеров как единого целого  

*Все команды выполняются строго из папки, где лежит docker-compose.yml*   

**sudo docker compose up -d** — поднять всю инфраструктуру (все сервисы) в фоне.   
**sudo docker compose ps** — чекнуть статус контейнеров, запущенных через Compose.  
**sudo docker compose logs** — смотреть живые логи всех сервисов одновременно.  
**sudo docker compose down** — полностью остановить и удалить все контейнеры проекта.  

флаг `-d` (detach) - позволяет запустить докер как фоновый процесс и не блокировать консоль

##  Сети в Dockrt и Порты(ports vs expose)  

**Короче есть две директивы - ports и expose.**   

**Ports** - это такая хуйня которая позволяет нам в файле с расширением .yml написать что то типо ports: - "80:80" (то есть <порт на хосте><порт внутри контейнера>), тем самым докер берет порт 80 внутри контейнера и *привязаывает* его к порту 80 моего ноутбука или утсройства с которого я работаю. Теперь любой тип который в моей вайфай сети(или вообще любой человек в интернете, если у сервера есть белый айпи) может вбить в браузере `http://<айпи>:<порт>` и попасть внтурь моего контейнера.  
    Как можно было заметить тут в примере выше трафик идет по протоколу HTTP — это небезопасная тема, так как данные передаются открытым текстом и любой тип в той же Wi-Fi сети через Wireshark может перехватить и легко ПРОЧИТАТЬ твои пароли.   
    Если использовать HTTPS (для этого нужен домен, например https://resale-bot.uz), то хакер ВСЁ ЕЩЕ сможет перехватить пакеты, но они будут намертво зашифрованы, и прочитать их он не сможет.  

**expose** — это директива, которая открывает порты ТОЛЬКО внутри внутренней сети Docker для других контейнеров. На моем ноуте/сервере снаружи этот порт доступен не будет. Если написать `expose: - "5432"`, порт базы будет виден только боту, который крутится в соседнем контейнере.  

*Если вкратце:* **ports** юзаем для сервисов, которые должны видеть обычные люди из интернета (например, сайт или веб-админка). А **expose** (или вообще ничего, так как Compose умный и сам связывает контейнеры) — для баз данных (Postgres, Redis), чтобы к ним никто не мог подобраться снаружи сервера. Управлять базой админу в таком случае нужно либо проваливаясь внутрь контейнера через `docker exec`, либо через админку, спрятанную за реверс-прокси.
